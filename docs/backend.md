# Backend setup

This is the operator runbook for the backend. The application code itself
lives in `prisma/`, `src/lib/`, and the Next.js route handlers under
`src/app/`. This doc tells you what to do once when bringing up a new
environment (your laptop, a CI runner, or production).

## Status

Phase 1 complete: **Prisma schema, Zod validators, server-authoritative
coin economy, seed data**. Pages still read from hardcoded arrays — they
will be switched over to the DB in a follow-up commit, once the schema is
migrated against a real Postgres URL.

What is **not** wired yet, in order of next priority:

1. Page reads → DB (`src/app/{kind,admin,shop,ouder}/page.tsx`)
2. Clerk auth (signup/signin, household + parent rows, role gating)
3. Stripe (3 subscriptions + Customer Portal + webhook with idempotency)
4. Quiz play route (`/kind/spelen/[quizId]`) backed by `Session` table
5. Admin quiz CRUD with Zod-validated payloads + AuditLog
6. PostHog event capture for the 22 events from README §11
7. Playwright happy-path test (landing → signup → trial → quiz play)

## 1. Get a Postgres URL

Free tier is plenty for dev.

- **Neon** (recommended — instant branching for migrations): https://neon.tech →
  Create project → copy the **Pooled** connection string.
- **Supabase**: Project settings → Database → Connection string (URI).
- **Local Postgres**: any 14+ install works.

Paste it into `.env.local` (create the file if it doesn't exist) — the
key already exists in `.env.example`:

```
DATABASE_URL=postgresql://user:pass@host/dbname?sslmode=require
```

> Do **not** commit `.env.local`. The `.env*` pattern in `.gitignore`
> already excludes it.

## 2. Generate the client + run migrations

```bash
# Generates the typed client at src/generated/prisma/
npm run db:generate

# Creates the schema in your DB and applies pending migrations.
# First run will prompt for a migration name — call it "init".
npm run db:migrate
```

The Prisma client is generated into `src/generated/prisma/` (gitignored).
Always import it via `@/lib/db`, never directly from the generated folder.

## 3. Seed the database

```bash
npm run db:seed
```

Inserts:

- 1 household (NL region, 14-day trial active)
- 1 parent (role ADMIN, placeholder Clerk userId)
- 1 kid (Sara, groep 5, 428 coins) with a `KidConsent` record
- 5 published quizzes — one per `GameType` (MC, CATAPULT, MATCH, DRAG_ORDER, TYPE)
- 10 valid questions per quiz, all run through Zod validators before insert
- 15 workbook SKUs across 4 groep buckets
- 3 bundles
- 3 subscription SKUs (Maandelijks, Jaarlijks, Gezinsabonnement)

The script is idempotent — re-running it updates rows instead of duplicating
them.

## 4. Inspect

```bash
npm run db:studio
```

Opens Prisma Studio on http://localhost:5555 — a browsable view of the DB.

## 5. Run the tests

```bash
npm test
```

Coverage of the backend code so far:

- `src/lib/coins.test.ts` — coin math, streak multiplier, completion bonus
- `src/lib/quiz-schemas.test.ts` — payload validation per gameType, server-side grading

No DB is required for the unit tests; they're pure-function tests.

## Layout

```
prisma/
├── schema.prisma            # source of truth for the data model
├── seed.ts                  # idempotent seed data
└── migrations/              # generated by `db:migrate`, commit these
src/
├── generated/prisma/        # generated client, gitignored
└── lib/
    ├── db.ts                # PrismaClient singleton (use this)
    ├── quiz-schemas.ts      # Zod validators + server-side grading
    └── coins.ts             # server-authoritative coin economy
```

## Resetting

If you want to throw away all data (dev only):

```bash
npm run db:reset
```

This drops the schema, re-creates it, runs migrations, and re-seeds. Don't
do this against a database with real users on it.

## Production deploys

- Use `db:deploy` (not `db:migrate`) — it doesn't prompt and doesn't shadow
  the schema. Run it as a build step on Vercel.
- Set `DATABASE_URL` to a **non-pooled** connection for migrations, then
  switch to a pooled URL for the running app. Neon supports both URLs
  for the same DB.
